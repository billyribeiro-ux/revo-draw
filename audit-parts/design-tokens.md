## Audit: design-tokens

Scope: exact value parity of the `.x-web` token remap against Excalidraw's light theme (`theme.scss` + `variables.module.scss`). The web build remaps LayoutForge token NAMES to Excalidraw VALUES under `.x-web`; the Tauri build keeps the original OKLCH `:root` set (by-design, not audited for parity).

Method: each `.x-web` value in `src/lib/ui/excalidraw-web.css` was cross-checked against its cited Excalidraw source. Most values are lifted verbatim and are correct. The findings below are the real residual mismatches plus consumer-side mismatches where a remapped token is used in a context Excalidraw fills with a *different* token.

### Matchable findings

### By-design divergences (do NOT fix)

- **Tauri `:root` OKLCH token set** (`tokens.css`): the entire warm-paper/indigo OKLCH palette, custom radii (3/5/7/11/16px), custom shadows, motion tokens, and chrome dimensions (`--titlebar-h`, `--toolrail-h`, `--panel-w`, `--statusbar-h`, `--traffic-gutter`) are LayoutForge's native desktop identity. Only the `.x-web` build is contracted to match Excalidraw; the Tauri theme is intentionally a different product.
- **Canvas grid colors / canvas-bg** under `.x-web` are set to plain white + faint gray (`#ffffff`, `#ebebeb`, `#d6d6d6`) matching Excalidraw defaults; the semantic-layout grid behavior is a different product concern and not a token-parity issue.
- **`--guide: #ff6b6b`** (snapping/alignment guide) has no Excalidraw equivalent token — Excalidraw's alignment guides are a separate visual system; this is a LayoutForge feature color, by design.
- **Dark theme** (`theme.scss:179-280`): not implemented in `.x-web` (light-only). Out of scope for this axis unless a dark mode is added.
- **Font stack**: `.x-web` uses self-hosted Assistant matching Excalidraw's `--ui-font`; the Tauri build uses Inter — intentional per-build divergence.
- Logo/wordmark colors (`--color-logo-icon`, `--color-logo-text: #190064`) are intentionally NOT reproduced (no Excalidraw branding).
