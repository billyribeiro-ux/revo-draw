## Audit: dialogs-a11y

Axis focus: modal/dialog/overlay patterns — focus trap, Escape, backdrop, ARIA roles, entrance animation — plus general a11y gaps. Files audited: `IconPicker.svelte`, `ContextMenu.svelte`, `WelcomeScreen.svelte` vs Excalidraw `Dialog.tsx`, `Modal.tsx`, `Dialog.scss`, `Modal.scss`, `ContextMenu.scss`.

### Matchable findings

| Title | Severity | Our ref | Excal ref | Proposed fix |
|---|---|---|---|---|
| IconPicker dialog has no focus trap | a11y | `IconPicker.svelte:96` (the `role="dialog" aria-modal="true"` element has no Tab/Shift+Tab handling) | `Dialog.tsx:70-93` (Tab cycles focus within `queryFocusableElements(islandNode)`, wrapping at both ends) | Add a `keydown` handler on the `.picker` element that, on `Tab`, queries focusable elements inside the picker and wraps: if focus is on the last and `!shiftKey`, focus the first and `preventDefault()`; if on the first and `shiftKey`, focus the last and `preventDefault()`. With `aria-modal="true"` declared, focus currently escapes to the canvas behind the modal — a real keyboard-trap parity gap. |
| IconPicker does not restore focus on close | a11y | `IconPicker.svelte:81-83` (focuses search input on open; nothing captures/restores the previously-focused element) | `Dialog.tsx:52` (`useState(document.activeElement)`) + `Dialog.tsx:99-104` (`(lastActiveElement as HTMLElement).focus()` in `onClose`) | Capture `document.activeElement` when `open` flips true, and in `onClose` (after the parent hides the dialog) call `.focus()` on the saved element. Without this, dismissing the picker drops keyboard focus to `<body>`. |
| Modal entrance animation missing (backdrop fade + content scale-in) | visual | `IconPicker.svelte:135-157` (`.backdrop` / `.picker` appear instantly) | `Modal.scss` `Modal__background__fade-in` (`opacity 0→1, 0.1s linear`) and `Modal__content_fade-in` (`opacity 0→1` + `transform: scale(0.9)→scale(1)`, `0.025s ease-out`) | Add the two keyframe animations: backdrop `opacity 0→1` over `0.1s linear forwards`; picker `opacity 0→1` + `transform: scale(0.9)→scale(1)` over ~`0.025s ease-out forwards` (or a slightly longer easing for a desktop app). Respect reduced-motion / Excalidraw's `animations-disabled` equivalent by gating behind `@media (prefers-reduced-motion: no-preference)`. |
| IconPicker uses `aria-label` instead of associating the visible title | a11y | `IconPicker.svelte:96` (`aria-label="Icon picker"`; the header has no `<h2>`/title element) | `Dialog.tsx:111` (`labelledBy="dialog-title"`) + `Dialog.tsx:117-121` (`<h2 id="...-dialog-title" class="Dialog__title">`) | Excalidraw labels its dialog via a visible `<h2>` referenced by `aria-labelledby`. The picker has no visible title at all; add an `<h2 id="iconpicker-title">Icon picker</h2>` in the header and switch the dialog to `aria-labelledby="iconpicker-title"` for parity (visible heading + programmatic label). Lower priority than the trap/restore gaps. |
| Close button has no hover/active color transition states matching Dialog__close | cosmetic | `IconPicker.svelte:189-200` (`.close` has `:hover` background only; no distinct icon color states) | `Dialog.scss:18-35` (`.Dialog__close` color `--color-gray-40`, `:hover` `--color-gray-60`, `:active` back to `--color-gray-40`) | Excalidraw's close affordance shifts icon color on hover/active rather than adding a background. Match by setting the close icon color from `--ink-soft` → `--ink` on `:hover` (current background-only hover is a divergence, though defensible for this app's chrome). |
| ContextMenu separator color does not match Excalidraw's gray-5 token | cosmetic | `ContextMenu.svelte:193-197` (`.sep` `border-top: 1px solid var(--line)`) | `ContextMenu.scss` `.context-menu-item-separator { border-top: 1px solid $color-gray-5 }` | Excalidraw separators use a fixed light-gray (`$color-gray-5`). Confirm `--line` resolves to an equivalently subtle hairline; if it is darker/more saturated than gray-5, align the token. Tokenized divergence — only fix if visibly mismatched. |
| ContextMenu separator vertical margin differs | cosmetic | `ContextMenu.svelte:196` (`margin: 4px 0`) | `ContextMenu.scss` `.context-menu { padding: 0.5rem 0 }` with separators having no extra margin (item padding `0.25rem 1rem`) | Excalidraw separators sit flush (no extra margin) inside `0.5rem` list padding; ours adds `4px 0`. Drop the separator margin to match the tighter Excalidraw rhythm, or verify the net spacing matches. Minor. |

### By-design divergences (do NOT fix)

- **No portal / `excalidraw-modal-container`** (`Modal.tsx:23,43-66`, `Modal.scss`). Excalidraw renders modals through `createPortal` into a dedicated container with a `--zIndex-modal` layer. This app is a single full-bleed canvas SPA; the IconPicker's `position: fixed` + `z-index: 40/41` is the intended simpler equivalent. Do not introduce a portal system.
- **No `Dialog--fullscreen` / phone form-factor branch** (`Dialog.tsx:54,108-110,122-132`, `Modal.scss` `.Dialog--fullscreen`). Excalidraw's fullscreen dialog + always-rendered close button exists for mobile/phone form factors. This is a desktop (Tauri/macOS) single-user app — mobile layouts are explicitly out of scope per project guide. Keep the centered desktop picker.
- **WelcomeScreen branding** (`WelcomeScreen.svelte:50-59`). LayoutForge uses its own mark/wordmark; Excalidraw's logo/wordmark is trademarked and must not be reproduced. The hand-drawn hint arrow SVG paths are reused (geometry, not branding) and that is intentional.
- **WelcomeScreen menu contents** (`WelcomeScreen.svelte:61-72`: Open / Browse library). These map to LayoutForge's product surface, not Excalidraw's (which links to its own docs/socials/load-scene). Different product — by design.
- **ContextMenu has no checkmark/checkbox item variant** (`ContextMenu.scss` `.checkmark::before`). Excalidraw's context menu supports toggle items with a `\2713` glyph; LayoutForge's `CtxItem` union (`item` | `separator`) deliberately has no toggle items. Not a gap unless toggle items are later added.
