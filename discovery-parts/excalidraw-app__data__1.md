### excalidraw-app/data/tabSync.ts

Purpose: Cross-tab synchronization helper that tracks per-storage-key version timestamps in both an in-memory map and `localStorage`, so a tab can detect when another tab has written newer data to browser storage.

- `LOCAL_STATE_VERSIONS` (module-level `const`, L5-L8): An in-memory record mapping two storage keys (`STORAGE_KEYS.VERSION_DATA_STATE` and `STORAGE_KEYS.VERSION_FILES`) to numeric version timestamps, both initialized to `-1`. This object represents *this tab's* view of when it last persisted each piece of state. Side effect: mutated in place by `updateBrowserStateVersion` and `resetBrowserStateVersions`. Invariant: keys are exactly the two storage keys; values are millisecond epoch timestamps or the sentinel `-1` ("never saved by this tab").
- `BrowserStateTypes` (type alias, L10): `keyof typeof LOCAL_STATE_VERSIONS` — the union of the two valid storage-key strings. Types-only; constrains the `type` parameter of the three exported functions.
- `isBrowserStorageStateNewer = (type: BrowserStateTypes) => boolean` (L12-L15): Reads the persisted timestamp for `type` from `localStorage.getItem(type)` (defaulting the raw string to `"-1"` before `JSON.parse`), and returns `true` when the storage timestamp is strictly greater than this tab's in-memory `LOCAL_STATE_VERSIONS[type]`. Output: boolean indicating another tab (or a prior session) wrote newer data than this tab last saw. Notable detail: comparison is strict `>`, so equal timestamps are treated as "not newer"; relies on `JSON.parse` of a numeric string rather than `parseInt` (handles the `"-1"` fallback and any JSON-stringified number written by `updateBrowserStateVersion`). No side effects.
- `updateBrowserStateVersion = (type: BrowserStateTypes) => void` (L17-L25): Captures `Date.now()`, writes it (JSON-stringified) into `localStorage` under `type`, and updates the in-memory `LOCAL_STATE_VERSIONS[type]` to the same value. Side effects: writes `localStorage` and mutates the module-level map. Errors from `localStorage.setItem` (e.g. quota/availability) are caught and logged via `console.error` (note: the log string contains a typo "verison"); the in-memory map is *not* updated if the `setItem` throws, because the mutation occurs after the write in the try block. Invariant: after success, the in-memory and persisted versions are equal, so an immediate `isBrowserStorageStateNewer` returns `false` for this tab.
- `resetBrowserStateVersions = () => void` (L27-L39): Iterates over every key of `LOCAL_STATE_VERSIONS` (cast to `BrowserStateTypes[]`), writing the sentinel `-1` to `localStorage` and resetting the in-memory value to `-1` for each. Used to force every key back to the "never saved" baseline (e.g. on logout/reset) so the next storage read is considered newer. Side effects: writes `localStorage` and mutates the map for all keys. Errors are caught and logged via `console.error` (same "verison" typo). Invariant: all tracked keys end at `-1` on success.

Performance/coordination note: this is the lightweight backbone of Excalidraw's multi-tab consistency. It does not use the `storage` event; instead other modules poll `isBrowserStorageStateNewer` to decide whether to re-read scene/file data, keeping the comparison to a single `localStorage.getItem` + `JSON.parse` per check. The `-1` initial value guarantees a fresh tab always sees existing storage as "newer" on first comparison.

### excalidraw-app/data/TTDStorage.ts

Purpose: An IndexedDB-backed persistence adapter (via `idb-keyval`) for storing and retrieving the user's Text-To-Diagram (TTD) chat history.

- `TTDIndexedDBAdapter` (exported class, L11-L51): Static-only class implementing the `TTDPersistenceAdapter` interface (per the doc comment) for persisting `SavedChats`. Has no instances/constructor; all members are `static`.
  - `private static idb_name = STORAGE_KEYS.IDB_TTD_CHATS` (L13): The base IndexedDB database name, sourced from app constants.
  - `private static key = "ttdChats"` (L15): The fixed entry key under which the entire chats array is stored within the store.
  - `private static store = createStore(\`${idb_name}-db\`, \`${idb_name}-store\`)` (L17-L20): An `idb-keyval` store handle created once at class-definition (module-load) time, using `<idb_name>-db` as the database name and `<idb_name>-store` as the object-store name. Side effect: opens/creates the IndexedDB database lazily through idb-keyval. Invariant: a single shared store handle is reused by all reads/writes.
  - `static async loadChats(): Promise<SavedChats>` (L26-L37): Calls `idb-keyval`'s `get<SavedChats>(key, store)` to fetch the persisted chats. Returns the data, or `[]` when nothing is stored (`data || []`). On any error it logs a warning via `console.warn` and returns `[]` — a non-throwing, fail-soft read so the UI always receives a valid array. Output: a `SavedChats` array (possibly empty).
  - `static async saveChats(chats: SavedChats): Promise<void>` (L43-L50): Calls `idb-keyval`'s `set(key, chats, store)` to persist the entire chats array under the fixed key. On error it logs a warning via `console.warn` *and re-throws* — unlike `loadChats`, writes are not silently swallowed, so callers can react to a failed save. Side effect: writes to IndexedDB. Invariant: overwrites the whole array each call (no incremental/merge logic).

Notes: `SavedChats` is imported as a type from `@excalidraw/excalidraw/components/TTDDialog/types`; no geometry, coordinate-space, or numeric math is involved in this file. The asymmetry between `loadChats` (returns `[]` on error) and `saveChats` (re-throws on error) is the key behavioral invariant to preserve in any reimplementation.
