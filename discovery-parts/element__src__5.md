### packages/element/src/sortElements.ts

Reorders an element array so grouped elements are contiguous (defragmented by group, first-occurrence order preserved) and bound text elements sit immediately after their containers, without losing any elements.

- `defragmentGroups(elements: readonly ExcalidrawElement[])` (L5-L54) — Internal. Recursively clusters elements by group at each nesting level so members of the same group are contiguous. Inner helpers: `groupIdAtLevel(element, level)` indexes `groupIds` from the END (`length - level - 1`) because `groupIds` is stored innermost-first so level 0 = outermost (L6-L8, L42-L43); `orderLevel(levelElements, level)` builds `slots` of either loose elements or group-id placeholders preserving first-occurrence order, buckets members by group id, then `flatMap`s expanding each group placeholder by recursing into `orderLevel(..., level+1)` (L10-L40). If the result count differs from input it logs `console.error` and returns the original array unchanged (data-loss guard, L48-L51). Side effect: `console.error` on the loss path.
- `normalizeBoundElementsOrder(elements: readonly ExcalidrawElement[])` (L65-L113) — Internal. Uses a `Set<ExcalidrawElement>` to dedupe-and-order: for each element with `boundElements`, adds the container then immediately adds each bound child of `type === "text"` looked up via `arrayToMap` (L77-L86); a text element that has a `containerId` whose container lists it is SKIPPED (it was already emitted after the container, L90-L98); otherwise the element is added (L100). Prefers the container's original z-index (moves bound text after container). Same data-loss guard: on size mismatch logs `console.error` and returns original (L105-L110). Side effect: `console.error` on loss path.
- `normalizeElementOrder(elements: readonly ExcalidrawElement[])` (L115-L119) — Exported. Composes the two: `normalizeBoundElementsOrder(defragmentGroups(elements))`. Pure aside from the internal console.error guards. Parity note: ordering rule = defragment groups first, THEN pull bound text next to containers; both steps are non-destructive (bail to original on any element loss).

### packages/element/src/textWrapping.ts

Browser-like soft-wrapping engine: builds Unicode-aware line-break-opportunity regexes (latin, CJK, emoji), tokenizes hard lines, and reflows tokens/words into visual lines with source-offset metadata.

- `cachedCjkRegex` / `cachedLineBreakRegex` / `cachedEmojiRegex` (module `let`s, L23-L25) — Lazy regex caches. Side effect: mutated on first use.
- `containsCJK(text)` (exported, L30-L36) — Lazily builds a CJK character class from `Object.values(CJK)` and tests `text`. Side effect: populates `cachedCjkRegex`.
- `getLineBreakRegex()` (L38-L48) — Internal. Lazily builds the line-break regex, trying `getLineBreakRegexAdvanced` (lookbehind-based) and falling back to `getLineBreakRegexSimple` on throw (Safari < 16.4 lacks lookbehind, L40-L44). Side effect: caches.
- `getEmojiRegex()` (L50-L56) — Internal. Lazily caches `getEmojiRegexUnicode()`.
- `COMMON` (const, L61-L84) — Break-rule char classes: `WHITESPACE`, `HYPHEN`, `OPENING`, `CLOSING` (western punctuation). Documents break-before/after semantics.
- `CJK` (const, L89-L118) — `CHAR` (Han/Hiragana/Katakana/Hangul + symbols), `OPENING`, `CLOSING`, `CURRENCY` (breaks before, not after). Parity note: CJK chars break before AND after unless paired with opening/closing punctuation.
- `EMOJI` (const, L120-L127) — `FLAG` (`\p{RI}\p{RI}`), `JOINER` (modifiers/VS/keycap/tags), `ZWJ`, `ANY`, `MOST` (`Extended_Pictographic`/`Emoji_Presentation`).
- `getLineBreakRegexSimple()` (L138-L142) — Internal fallback: OR of emoji regex and `Break.On(HYPHEN, WHITESPACE, CJK.CHAR)`. No advanced CJK rules; covers latin.
- `getLineBreakRegexAdvanced()` (L151-L169) — Internal. The full rule set: emoji; break before whitespace, after whitespace/hyphen; CJK char/currency before (not preceded by opening); CJK char after (not followed by hyphen/closing); opening/closing punctuation pairing rules (L152-L168). Uses lookbehind. Parity note: this is the authoritative break-opportunity grammar.
- `getEmojiRegexUnicode()` (L194-L206) — Internal. Builds the grouped emoji regex matching flags, ZWJ sequences, skin tones, VS, keycaps, tag sequences (per Unicode TR51 with documented substitutions, L171-L206).
- `Regex` (util object, L211-L239) — Regex string-composition helpers: `build(str)` (→ `new RegExp(str,"u")`), `join` (concat sources), `and` (concat), `or` (`|` join), `group` (`(...)`), `class` (`[...]`). All return `/u` regexes. Pure.
- `Break` (util object, L245-L373) — Human-readable lookahead/lookbehind builders for break opportunities. `On` (capture group), `Before` (`(?=...)`), `After` (`(?<=...)`), `BeforeMany`/`AfterMany` (greedy class boundaries), `NotBefore` (`(?!...)`), `NotAfter` (`(?<!...)`), each returning a chainable builder (type omits the opposite-direction method). `Chain(rootBuilder)` exposes `Build`, `PreceededBy` [sic], `FollowedBy`, `NotPrecededBy`, `NotFollowedBy` which `Regex.and` additional lookarounds onto the root (L319-L372). Pure composition. Parity note: the chaining encodes the "X but not preceded/followed by Y" grammar used in the advanced regex.
- `parseTokens(line)` (exported, L382-L389) — NFC-normalizes the line, `split`s on the break-opportunity regex, filters out empty strings. Returns breakable tokens. Pure. Parity note: NFC normalization means decomposed graphemes are treated as composed; callers needing exact source offsets must pre-NFC the input (offsets can shift otherwise).
- `wrapText(text, font, maxWidth): string` (exported, L397-L405) — Convenience over `getWrappedTextLines`: maps to `.text` and joins with `\n`. Pure (given stable measurement).
- `type WrappedTextLine = { text; start; end }` (L414-L418) — A visual line with end-exclusive code-unit offsets into the ORIGINAL text, excluding synthetic soft breaks; on trim, `end` points at the last rendered char.
- `getHardLineBreaks(text): WrappedTextLine[]` (L423-L438) — Internal. Splits on `\n` only, tracking running `offset` (`+1` for the consumed newline) to preserve source offsets. Pure. Used as the fallback when `maxWidth` is non-finite/negative.
- `getWrappedTextLines(text, font, maxWidth): WrappedTextLine[]` (exported, L446-L478) — Main wrapping pipeline. Guards against non-finite/negative `maxWidth` by returning hard-line breaks (prevents infinite loop, L451-L456). For each hard line: if it fits (`getLineWidth <= maxWidth`) emit as-is, else `wrapLine`; advances `offset` by `line.length + 1` (newline) (L461-L475). Pure. Parity note: the maxWidth finiteness guard is essential — bad geometry must not loop forever.
- `wrapLine(line, font, maxWidth, lineStart): WrappedTextLine[]` (internal, L486-L573) — Reflows one hard line's tokens into visual lines. Tracks current line text/start/end/width and per-token source offset. Uses cached single-codepoint width via `charWidth.calculate` when the token is a single char (kerning ignored), else `getLineWidth(testLine)` (L509-L512). Whitespace tokens or fitting tokens extend the current line (L515-L525). If the current line is empty and the lone token exceeds width, `wrapWord` splits it and the trailing fragment becomes the new current line (L527-L545). Otherwise push the current line (trimmed at the soft break via `trimLineEndAtSoftBreak`) and reset WITHOUT consuming the token (L546-L557). Trailing line is pushed via `trimLine` (L561-L570). Pure (given measurement). Parity note: the "don't consume the token on overflow" reset (L552) is why the loop uses `!currentLine` checks; whitespace is allowed to overflow to mimic browser trailing-space behavior.
- `wrapWord(word, font, maxWidth, wordStart): WrappedTextLine[]` (internal, L578-L647) — Breaks a single over-long token char-by-char. Multi-codepoint emojis are returned whole (not split) if `getEmojiRegex().test(word)` (L585-L593). Asserts no whitespace via `satisfiesWordInvariant` (L595). Iterates `Array.from(word)` (code-point aware), accumulating chars using cached `charWidth.calculate` until exceeding maxWidth, then flushing (L598-L644). Pure. Parity note: char iteration uses `Array.from` (code points), char offsets use `char.length` (code units) — both matter for surrogate pairs.
- `trimLine(line, start, end, font, maxWidth): WrappedTextLine` (internal, L655-L698) — For the trailing visual line: if it doesn't exceed maxWidth, returns as-is (L662-L670); else splits trailing whitespace via `/^(.+?)(\s+)$/`, then greedily re-adds whitespace chars while they still fit (`charWidth.calculate`), adjusting `end` by the removed length (L673-L697). Pure. Parity note: trailing whitespace is partially preserved up to maxWidth — mimics browser rendering.
- `trimLineEndAtSoftBreak(line, start, end): WrappedTextLine` (internal, L705-L717) — For internal soft-break boundaries: `trimEnd()`s the line and shifts `end` back by the removed length. Pure. Parity note: soft-break lines drop ALL trailing whitespace (unlike `trimLine`).
- `isSingleCharacter(maybeSingleCharacter)` (internal, L724-L729) — True when `codePointAt(0)` is defined and `codePointAt(1)` is undefined (single code point; handles é/中, deliberately not multi-codepoint emoji). Pure.
- `satisfiesWordInvariant(word)` (internal, L734-L740) — In test/dev envs throws if the word contains whitespace. Side effect: throws in dev/test only.
