## Cluster: excalidraw__renderer__0

This cluster covers the Canvas-2D rendering layer of Excalidraw: the static element canvas, the interactive overlay canvas (selection borders, transform/crop handles, linear-element point handles, binding highlights), the in-progress "new element" canvas, snap-guide rendering, a shared animation scheduler, low-level canvas helpers, and a rounded-rect path primitive. All coordinate work happens in world-space after `context.scale(zoom)` and `context.translate(scrollX, scrollY)`; line widths and handle sizes are divided by `zoom.value` so they stay screen-constant.

### packages/excalidraw/renderer/animation.ts

Purpose: A static, key-keyed animation scheduler that drives stateful per-frame callbacks via `requestAnimationFrame` (or `setTimeout` when render-throttling is disabled), used by the interactive scene's animated binding highlight.

- `type Animation<R extends object> = (params: { deltaTime: number; state?: R }) => R | null | undefined` (L3-L6) — the callback contract: receives the elapsed `deltaTime` (ms) and prior `state`, returns next state to continue or a falsy value to stop. Returning `null`/`undefined` removes the animation from the loop.
- `class AnimationController` (L8-L126) — singleton-style controller; all members are `static`. Holds two static fields:
  - `scheduledFrame` (L9-L12): tagged union `{id, type:"raf"} | {id, type:"timeout"} | null` tracking the single pending frame across ALL animations (one shared frame, not one per animation).
  - `animations` (L13-L20): `Map<string, {animation, lastTime, state}>` keyed by caller-supplied string key, so duplicate `start()` calls for the same key are idempotent.
  - `static start<R extends object>(key: string, animation: Animation<R>)` (L22-L41) — if `key` already registered, no-op (L23-25). Otherwise invokes the animation once with `deltaTime:0, state:undefined` to get an initial state (L27-30); only if that initial state is truthy does it register the entry and schedule the next frame (L32-40). Invariant: an animation that returns falsy on first call never enters the map.
  - `private static scheduleNextFrame()` (L43-L59) — guards against double-scheduling (L44-46); picks `requestAnimationFrame(tick)` when `isRenderThrottlingEnabled()` else `setTimeout(tick, 0)` (L48-58). Performance note: rAF coalesces to display refresh; the timeout path runs as fast as the event loop allows.
  - `private static cancelScheduledFrame()` (L61-L73) — cancels the pending rAF/timeout by tag and nulls `scheduledFrame`.
  - `private static cancelScheduledFrameIfIdle(): boolean` (L75-L82) — returns false (and does nothing) if any animations remain; otherwise cancels the frame and returns true. Used to short-circuit the tick loop when the map empties.
  - `private static tick()` (L84-L116) — the loop body. Clears `scheduledFrame` (L85), then for each registered animation computes `deltaTime = lastTime===0 ? 0 : now - lastTime` using `performance.now()` per-entry (L89-91), runs the callback, deletes entries whose result is falsy (and early-returns if that empties the map, L98-103), otherwise updates `lastTime`/`state` (L104-107). After the loop, re-schedules a frame unless idle (L110-114). Side effect: mutates the static map in place.
  - `static running(key: string): boolean` (L118-L120) — membership check on the map.
  - `static cancel(key: string)` (L122-L125) — removes a key and cancels the shared frame if nothing is left.

Notable: `lastTime===0` sentinel means the first real frame after registration also reports `deltaTime 0` (the initial probe call already consumed time 0). `performance.now()` is sampled inside the per-entry loop, so multiple animations in one tick can see slightly different `now` values.

### packages/excalidraw/renderer/helpers.ts

Purpose: Low-level Canvas-2D primitives shared by every renderer — circle fill, DPR-normalized dimension math, canvas bootstrap (transform reset + background paint), and a rotated stroke-rect.

- `fillCircle(context, cx, cy, radius, stroke, fill = true)` (L6-L22) — `beginPath` + full `arc(0..2π)`; fills if `fill`, strokes if `stroke`. No `save/restore` — caller owns fillStyle/strokeStyle. Used for transform-handle rotation circles and linear-point handles.
- `getNormalizedCanvasDimensions(canvas, scale): [number, number]` (L24-L30) — returns `[canvas.width/scale, canvas.height/scale]`. Converts the DPR-scaled backing-store size to CSS/logical pixels so downstream math works in logical space. Invariant: `scale` here is `window.devicePixelRatio` (the canvas backing scale), NOT the camera zoom.
- `bootstrapCanvas({canvas, scale, normalizedWidth, normalizedHeight, theme?, isExporting?, viewBackgroundColor?}): CanvasRenderingContext2D` (L32-L76) — resets the transform to identity via `setTransform(1,0,0,1,0,0)` then `scale(scale, scale)` to apply DPR (L51-52). Background handling (L55-73): if `viewBackgroundColor` is a string, detects transparency (literal `"transparent"`, 5-char `#RGBA`, 9-char `#RRGGBBA`, or `hsla(/rgba(` regex) and `clearRect`s first when transparent (L56-63); then fills the whole canvas with the color, applying `applyDarkModeFilter` when `theme===THEME.DARK` (L64-70). If no color string given, just `clearRect`s (L71-73). Returns the 2D context. Side effects: clears/paints the entire canvas and leaves the DPR scale applied (the only transform on entry to scene rendering).
- `strokeRectWithRotation_simple(context, x, y, width, height, cx, cy, angle, fill = false, radius = 0)` (L78-L106) — draws a rectangle rotated `angle` radians about pivot `(cx, cy)`. Algorithm: `save` → `translate(cx,cy)` → `rotate(angle)` → draw with coords shifted by `-cx,-cy` (so the rect is positioned relative to the rotated origin) → `restore` (L91-105). Optionally `fillRect` first (L94-96). If `radius` and `context.roundRect` exist, uses a rounded path + stroke; else `strokeRect` (L97-104). The `radius` param comment says it "should account for zoom" — callers pass `radius/zoom.value`. Coordinate-space detail: pivot rotation here is the canonical way Excalidraw renders rotated selection borders/handles.

### packages/excalidraw/renderer/roundRect.ts

Purpose: A single rounded-rectangle path primitive (manual quadratic-curve corners) used for scrollbars and as a fallback when the native `context.roundRect` is unavailable.

- `roundRect(context, x, y, width, height, radius, strokeColor?)` (L11-L41) — builds a rounded-rect path manually: `moveTo(x+radius, y)`, four straight `lineTo` edges joined by four `quadraticCurveTo` corners with the rectangle's actual corners as control points (L20-35), then `closePath`, `fill`, optionally set `strokeStyle`, and `stroke` (L36-40). Always both fills and strokes. Geometry note: corner control point is the true rect corner (e.g. `(x+width, y)`), giving a circular-ish quadratic fillet of `radius`. Attribution comment cites stackoverflow.com/a/3368118. Invariant: assumes `radius <= min(width,height)/2`; no clamping.

### packages/excalidraw/renderer/renderSnaps.ts

Purpose: Renders alignment/spacing snap guides (point lines, pointer lines, gap lines) on the interactive canvas, with zoom-invariant sizing and theme/zen-mode color logic.

Module constants (L8-L11): `SNAP_COLOR_LIGHT = "#ff6b6b"`, `SNAP_COLOR_DARK = "#ff0000"`, `SNAP_WIDTH = 1`, `SNAP_CROSS_SIZE = 2`.

- `renderSnaps(context, appState: InteractiveCanvasAppState)` (L13-L61) — early-return if no snap lines (L17-19). Chooses color: light/zen → light color, else dark color (the comment at L21-23 explains dark-mode color inversion is avoided in zen mode because only crosses are drawn) (L24-27). Computes `snapWidth = (zen ? SNAP_WIDTH*1.5 : SNAP_WIDTH)/zoom.value` (L29-31). `save` + `translate(scrollX, scrollY)` (L33-34), then dispatches per snap line by `type`: `"pointer"` → `drawPointerSnapLine`, `"gap"` → `drawGapLine`, `"points"` → `drawPointsSnapLine`, each setting `lineWidth`/`strokeStyle` first (L36-58). `restore` at L60.
- `drawPointsSnapLine(pointSnapLine, context, appState)` (L63-L78) — unless zen mode, draws a single line from first to last point (L68-73), then a cross at every point (L75-77).
- `drawPointerSnapLine(pointerSnapLine, context, appState)` (L80-L89) — always draws a cross at `points[0]`; unless zen mode also draws a line from `points[0]` to `points[1]`.
- `drawCross<Point>([x,y], appState, context)` (L91-L110) — draws an X: size = `(zen ? SNAP_CROSS_SIZE*1.5 : SNAP_CROSS_SIZE)/zoom.value` (L97-99); two diagonal strokes from `(x-size,y-size)→(x+size,y+size)` and `(x+size,y-size)→(x-size,y+size)` (L102-107). `save/restore` wrapped.
- `drawLine<Point>(from, to, context)` (L112-L121) — `beginPath` then two `lineTo` calls (note: starts with `lineTo(from)` not `moveTo` — relies on current path position being reset by `beginPath`, so effectively a degenerate first segment from the path origin; in practice the prior subpath is empty so it behaves as a move). Strokes the line.
- `drawGapLine<Point>(from, to, direction, appState, context)` (L123-L209) — renders the equal-gap indicator with end-caps and a doubled mid-tick. Sizes are zoom-invariant: `FULL = 8/zoom`, `HALF = FULL/2`, `QUARTER = FULL/4` (L136-138). For `"horizontal"` (L140-173): computes `halfPoint` at the x-midpoint (L141); draws cap (1) at `from`, two mid-ticks (3) offset by ±QUARTER around the half point, cap (4) at `to`, and the connecting line (2) — caps and connector are suppressed in zen mode, mid-ticks always drawn (L143-172). The `"vertical"` branch mirrors the geometry swapping x/y roles (L174-208). The ASCII diagram at L130-134 documents the four tick groups. Uses `pointFrom` from `@excalidraw/math` to build points.

### packages/excalidraw/renderer/interactiveScene.ts

Purpose: The interactive overlay canvas renderer — selection borders, transform/crop handles, linear-element point & midpoint handles, binding highlights (simple and animated "complex" variants), focus-point indicators, text bounding box, search-match highlights, remote cursors, and scrollbars.

- `renderElbowArrowMidPointHighlight(context, appState)` (L115-L131) — asserts `selectedLinearElement` and its `segmentMidPointHoveredCoords` are non-null via `invariant`, translates by scroll, and draws a `highlightPoint` at the hovered elbow segment midpoint.
- `renderLinearElementPointHighlight(context, appState, elementsMap: ElementsMap)` (L133-L165) — highlights the hovered linear point unless it's already selected-while-editing (L139-146) or the element is being dragged (L147-149). Resolves the element via `LinearElementEditor.getElement`, gets the global point at `hoverPointIndex` via `getPointAtIndexGlobalCoordinates`, translates by scroll and draws the highlight.
- `highlightPoint<Point>(point, context, appState)` (L167-L181) — fills a translucent purple circle (`rgba(105,101,219,0.4)`) of radius `POINT_HANDLE_SIZE/zoom.value` at the point. Zoom-invariant radius.
- `renderFocusPointHighlight(context, appState, focusPoint: GlobalPoint)` (L183-L194) — scroll-translated `highlightPoint` at an arrow binding focus point.
- `renderSingleLinearPoint<Point>(context, appState, point, radius, isSelected, isPhantomPoint, isOverlappingPoint)` (L196-L224) — draws one linear-element vertex handle. Stroke `#5e5ad8`; fill varies: selected → `rgba(134,131,226,0.9)`, phantom → `rgba(177,151,252,0.7)`, else white-ish (L205-212). Final radius is scaled: overlapping points get `radius * (editing?1.5:2)` else `radius`, divided by zoom (L218-220); fill enabled unless phantom; stroke enabled unless overlapping-and-unselected (L221-222).
- `renderBindingHighlightForBindableElement_simple(context, suggestedBinding, elementsMap, appState, pointerCoords)` (L226-L556) — the non-feature-flagged binding highlight. If the element sits in a frame, clips to the (rounded) frame rect first (L233-256). For `frame/magicframe` it strokes the frame outline in blue (L258-292). Otherwise it rotates the context about the element center (L294-300), translates to the element origin, sets a zoom-clamped `lineWidth` (`clamp(1.75, strokeWidth, 4)/max(0.25, zoom)`, L304-306) and blue stroke, then per shape: `ellipse` via `context.ellipse` (L313-326); `diamond` and default `rectanguloid` by deconstructing the element into line `segments` + bezier `curves` (`deconstructDiamondElement`/`deconstructRectanguloidElement`) and stroking each segment/curve with coords offset by `-element.x/-element.y` (L327-409). Then, if midpoint snapping is enabled and the element is frame-like/bindable (L416-555), it computes 4 cardinal midpoints (diamond uses `bezierEquation(curve,0.5)` of base corners rotated by `pointRotateRads`; others use RIGHT/BOTTOM/LEFT/TOP base points rotated about center) (L446-489), finds the closest midpoint to `pointerCoords` via a `reduce` (L491-512), and draws highlighted (blue) or shown (gray) midpoint dots of radius `4/zoom`, gated by `maxBindingDistance_simple(zoom)+strokeWidth/2` thresholds and elbow-arrow special-casing (L514-551). Heavy geometry/coordinate-space code — central to binding-UI parity.
- `renderBindingHighlightForBindableElement_complex(app, context, element, allElementsMap, appState, deltaTime, state?)` (L558-L888) — the animated (feature-flagged `COMPLEX_BINDINGS`) variant. Computes an `opacity` from a `BIND_MODE_TIMEOUT` countdown driven by `state.runtime` and `app.state.bindMode==="orbit"` (L567-573); uses an `offset = strokeWidth/2` to grow the outline (L574). Frame clipping (L576-597) and per-shape outline drawing mirror the simple variant but include `appState.scrollX/scrollY` in the center/origin translation (L633-643) and inflate ellipse/segment/curve geometry by `offset` (L653-757). Below the timeout it also draws an animated dashed center "snap area" ellipse with `lineDashOffset` driven by progress ratio (L766-808) and, if midpoint snapping enabled, clears cutout squares around midpoints and paints midpoint dots (L810-882). Returns `{runtime: (state?.runtime ?? 0) + deltaTime}` to advance the animation (L885-887). This is the only renderer function that returns animation state.
- `renderBindingHighlightForBindableElement(app, context, suggestedBinding, allElementsMap, appState, deltaTime, state?)` (L890-L931) — dispatcher: returns early if no binding; if `getFeatureFlag("COMPLEX_BINDINGS")` delegates to the complex variant (returning its state); otherwise `save`/`translate(scroll)`, builds `pointerCoords` from `app.lastPointerMoveCoords`, calls the simple variant, `restore`.
- `type ElementSelectionBorder` (L933-L945) — shape describing one selection box: `angle, x1,y1,x2,y2, selectionColors[], dashed?, cx,cy, activeEmbeddable, padding?`.
- `renderSelectionBorder(context, appState, elementProperties)` (L947-L1000) — draws a (possibly multi-color, possibly dashed) selection rectangle. Padding defaults to `DEFAULT_TRANSFORM_HANDLE_SPACING*2`, all sizes divided by zoom (`linePadding`, `lineWidth=8/zoom`, `spaceWidth=4/zoom`) (L967-972). For each color it sets a dash pattern that interleaves the N colors (each color's dash is offset by `(lineWidth+spaceWidth)*index` so multiple remote/local colors form a single multicolor dashed border) (L979-998), drawing via `strokeRectWithRotation_simple`. `activeEmbeddable` thickens the line to 4/zoom.
- `renderFrameHighlight(context, appState, frame, elementsMap)` (L1002-L1030) — strokes a blue (`rgb(0,118,255)`) rounded rect around a frame using `getElementAbsoluteCoords` and `strokeRectWithRotation_simple` with `FRAME_STYLE.radius/zoom`.
- `renderElementsBoxHighlight(context, appState, elements, config?)` (L1032-L1078) — draws box highlights for a set of elements: splits into ungrouped individuals vs grouped (L1039-1045); builds an `ElementSelectionBorder` per group (via `getCommonBounds` of `getElementsInGroup`) and per individual element, then renders each with `renderSelectionBorder`. Default color `rgb(0,118,255)`, optional dashed. Used for frame-drop and locked-element highlights.
- `renderLinearPointHandles(context, appState, element, elementsMap: RenderableElementsMap)` (L1080-L1203) — renders all vertex and midpoint handles for a selected linear element. Returns early if no `selectedLinearElement` (L1086-88). Radius is `POINT_HANDLE_SIZE` when editing else half (L1098-1100). Gets global point coords; for elbow arrows only renders first/last vertices (L1106-1108). Detects overlapping points (coincident within `2/zoom`) (L1110-1117), computes selection state including the polygon last-point-overlaps-first-point special case (L1119-1134), and calls `renderSingleLinearPoint` per vertex (L1136-1145). Then renders segment midpoints: for elbow arrows, midpoints of non-too-short segments as phantom points (L1148-1175); for others, `getEditorMidPoints` filtered, rendered as phantom points when editing or 2-point line (L1176-1200).
- `renderFocusPointConnectionLine(context, appState, fromPoint, toPoint)` (L1205-L1224) — dashed purple (`rgba(134,131,226,0.6)`) line from an arrow endpoint to its binding focus point; dash `4/zoom`, width `1/zoom`.
- `renderFocusPointCicle(context, appState, point, radius, isHovered)` (L1226-L1251) — [sic name] filled+stroked purple circle at the focus point, brighter fill when hovered; radius `radius/zoom`.
- `renderFocusPointIndicator({arrow, appState, type, context, elementsMap})` (L1253-L1343) — for an arrow's `"start"`/`"end"` binding, resolves the bound element and its global fixed focus point (`getGlobalFixedPointForBindableElement`), bails if the element is missing/deleted/not bindable or the focus point isn't visible (`isFocusPointVisible`) (L1266-1296). Optionally renders the hovered highlight (L1307-1312), then (unless the point is selected-and-dragging) draws the connection line from the arrow endpoint to the focus point plus the focus circle at `FOCUS_POINT_SIZE/1.5` (L1317-1342).
- `renderTransformHandles(context, renderConfig, appState, transformHandles: TransformHandles, angle): void` (L1345-L1386) — iterates handle keys; for each defined handle `[x,y,width,height]` sets `lineWidth=1/zoom` and `strokeStyle=selectionColor`. The `"rotation"` handle is drawn as a filled+stroked circle (`fillCircle`) (L1362-1364); other handles prefer native `context.roundRect` (radius `2/zoom`) else fall back to `strokeRectWithRotation_simple` with `fill:true` so the square is white-filled then stroked (L1365-1382).
- `renderCropHandles(context, renderConfig, appState, croppingElement: ExcalidrawImageElement, elementsMap): void` (L1388-L1491) — draws the four L-shaped image-crop corner brackets. Computes half-width/height from absolute coords plus a zoomed line width (`LINE_WIDTH=3`, `LINE_LENGTH=20`, both /zoom) (L1400-1416). Builds a hand-coded array of 4 corner descriptors, each holding the corner origin and two line segments (horizontal + vertical) whose lengths are clamped to the available half-dimension (L1423-1469). Each corner is drawn rotated about the element center `(cx,cy)` by `croppingElement.angle` (L1471-1488). Non-obvious geometry: the bracket arms point inward and are offset by half the line width so strokes don't overhang the image edge.
- `renderTextBox(text, context, appState, selectionColor)` (L1493-L1515) — dashed translucent (alpha 0.5) selection rect around a text element being edited; padding from `getTextBoxPadding(zoom)`, rotated about the text center, dash `[6/zoom, 4/zoom]`.
- `renderResetAutoResizeHandle(text, context, appState, selectionColor, formFactor)` (L1517-L1550) — draws the small "reset auto-resize" handle line (from `getTextAutoResizeHandle`) with rounded caps, alpha 0.5, width `1.5/zoom`; bails if no handle.
- `_renderInteractiveScene(config: InteractiveSceneRenderConfig): { scrollBars?, atLeastOneVisibleElement, elementsMap, animationState? }` (L1552-L2076) — THE main interactive-canvas pass. Sequence: bail if no canvas (L1571-73); compute normalized dims; `bootstrapCanvas` (no background — overlay is transparent) (L1581-86); `save` + `scale(zoom)` (L1589-90). Finds the currently-editing linear element from `visibleElements` (to avoid stale `ShapeCache` versions during collab, per the L1596-1598 comment) and renders its point handles (L1592-1616). Renders the marquee selection element (L1618-1630); the text auto-resize handle for an active non-auto-resize text element (L1632-1642); the editing-text bounding box (L1644-1656). Computes the binding highlight into `nextAnimationState.bindingHighlight` when binding is enabled, else clears it (L1658-1676). Frame highlight (L1678-1685), elements-box highlight (L1687-1689), active-locked highlight in gray dashed (L1691-1700). Re-renders linear point handles for a single editing linear selection (L1709-1720). Renders elbow/linear point hover highlights and arrow focus-point indicators for the selected linear element (L1722-1760). The big selection block (L1763-1985, gated on not multiElement/newElement/editing-linear): computes `showBoundingBox` via `hasBoundingBox`; for a single linear selection renders its handles; iterates ALL `elementsMap` values building per-element selection borders combining local selection color and remote-collaborator colors (skipping bound elbow arrows), with locked elements drawn gray dashed and image/cropping elements getting zero padding (L1791-1859); adds dashed group-bound selections for selected groups and the editing group (L1861-1887); renders all selection borders (L1889-1891). Then transform handles: single selection uses `getTransformHandles` (hidden in view mode / while text-editing / while cropping) and optional crop handles (L1897-1935); multi-selection (>1, not rotating, none locked) draws a dashed common-bounds rect and `getTransformHandlesFromCoords` at angle 0, omitting rotation when a frame is selected (L1936-1983). Search matches are painted as translucent yellow/orange rects, rotated about each matched element's center, with a frame-specific zoom factor (L1987-2028). `renderSnaps`, then `restore` to undo the zoom (L2030-2032). `renderRemoteCursors` is called AFTER restore (it manages its own transform) (L2034-2040). Scrollbars are drawn last via `getScrollBars` + `roundRect` when `renderScrollbars` (L2042-2068). Returns scrollbars, visibility flag, the elementsMap, and the next animation state (L2070-2075). Key coordinate detail: the entire body runs inside one `scale(zoom)`; nearly every nested draw additionally `translate(scrollX, scrollY)` so element world coords map to screen. Performance: builds selection borders by iterating the whole `elementsMap` (not just selected) to also catch remote selections.
- `renderInteractiveScene<U>(renderConfig): ReturnType<U>` (L2082-L2090) — public entry; calls `_renderInteractiveScene`, invokes `renderConfig.callback(ret)` (used by `Renderer` to push scrollbars/state back into the app), and returns the result cast to `U`.

### packages/excalidraw/renderer/renderNewElementScene.ts

Purpose: Renders the single element currently being created/dragged on its own dedicated canvas (separate from the static scene so the rest of the document isn't redrawn each pointermove), with optional frame clipping and rAF throttling.

- `_renderNewElementScene({canvas, rc, newElement, elementsMap, allElementsMap, scale, appState, renderConfig})` (L16-L85) — bails if no canvas. Normalizes dims, `bootstrapCanvas` (transparent overlay), `save`, `scale(zoom)` (L27-42). If `newElement` exists and isn't a `"selection"`: returns early (skipping render) if `isInvisiblySmallElement` (e.g. arrow below drag threshold) (L49-51); resolves a frame from `newElement.frameId` or `frameToHighlight` and, when frame rendering+clip are enabled and `shouldApplyFrameClip`, applies `frameClip` (L53-68); then `renderElement` draws the in-progress element (L70-78). If there's no element (or it's a selection), `clearRect`s the canvas (L79-81). `restore` (L83). Note: the early `return` at L50 leaves `context` un-restored — acceptable because the canvas is rebuilt each frame.
- `renderNewElementSceneThrottled` (L87-L91) — `throttleRAF`-wrapped `_renderNewElementScene`, so rapid pointer events coalesce to one draw per animation frame.
- `renderNewElementScene(renderConfig, throttle?)` (L93-L103) — public entry; routes to the throttled version when `throttle`, else calls `_renderNewElementScene` synchronously.

### packages/excalidraw/renderer/staticScene.ts

Purpose: The main non-interactive element canvas renderer — paints the background grid, all visible elements (with frame clipping, bound-text, and link icons), embeddables/iframes on top, and pending flowchart nodes; rAF-throttled.

Module-level: `GridLineColor` (L45-L54) — per-theme bold/regular grid colors, dark values precomputed via `applyDarkModeFilter`. `linkIconCanvasCache` (L160-L166) — module-level cache of two offscreen `LinkIconCanvas` (regular vs element link), each tagged with the `zoom` it was rendered at so it's only re-rasterized on zoom change (performance: avoids re-drawing the link bitmap every frame).

- `strokeGrid(context, gridSize, gridStep, scrollX, scrollY, zoom, theme, width, height)` (L56-L130) — draws the dotted/solid grid. Computes start offsets `(scroll % gridSize) - gridSize` (L69-70) and `actualGridSize = gridSize*zoom` (L72). Crispness hack (L78-84): only at `zoom===1` does it translate by 0.5px per-axis (and only on an axis whose offset isn't already fractional) so 1px lines land on pixel centers. Vertical loop (L87-107) and horizontal loop (L109-128): a line is `bold` when `gridStep>1` and its world coordinate is an exact multiple of `gridStep*gridSize`; non-bold regular lines are skipped entirely when `actualGridSize<10` (too dense to see); `lineWidth = min(1/zoom, isBold?4:1)`; bold lines are solid, regular lines use a computed `lineDash`. Coordinate-space: runs inside the caller's `scale(zoom)`, so positions are world coords. Lines extend `gridSize*2` beyond the viewport to avoid edge gaps.
- `frameClip(frame, context, renderConfig, appState)` (L132-L156) — sets a clipping region to a frame's bounds: translates to `frame.(x,y)+scroll`, builds a rounded (`FRAME_STYLE.radius/zoom`) or plain rect path, `clip()`, then translates back. Exported and reused by `renderNewElementScene`. Side effect: mutates the clip region (caller must `save`/`restore` around it).
- `type LinkIconCanvas = HTMLCanvasElement & { zoom: number }` (L158) — offscreen canvas tagged with its render zoom.
- `renderLinkIcon(element, context, appState, elementsMap)` (L168-L228) — draws the hyperlink badge for an element that has a `link` and isn't selected (L174). Computes the handle rect from `getLinkHandleFromCoords`, translates to the badge center + scroll, rotates by element angle (L175-186). Lazily rasterizes the icon into a DPR×zoom-scaled offscreen canvas (filled with the view background, then `EXTERNAL_LINK_IMG` or `ELEMENT_LINK_IMG` drawn) cached per zoom (L191-223). Draws the cached bitmap at `element.opacity/100` alpha (L224-225). Performance detail: the expensive image draw happens once per zoom level; subsequent frames just `drawImage` the cache.
- `_renderStaticScene({canvas, rc, elementsMap, allElementsMap, visibleElements, scale, appState, renderConfig})` (L229-L479) — THE main static pass. Bails if no canvas. Destructures `renderGrid=true, isExporting`. `bootstrapCanvas` WITH theme/background/isExporting (this canvas paints the document background) (L250-258); `scale(zoom)` (L261). Draws grid if enabled (passing viewport dims divided by zoom so the grid covers world-space) (L264-276). First pass (L278-296) computes `groupsToBeAddedToFrame` — group ids whose elements overlap the highlighted frame (so an entire group highlights together). Main element pass (L301-385): skips iframe-likes (rendered later) and bound text (rendered with its container); for each element, `save`, optionally `frameClip` to its target frame, `renderElement`, then render its bound text element, `restore`, and (unless exporting) `renderLinkIcon`; per-element try/catch logs element id+geometry on error. Embeddable pass (L388-461): renders iframe-likes on top, drawing a placeholder label for invalid/exporting embeds, with the same frame-clip logic. Flowchart pending-nodes pass (L463-478): renders `renderConfig.pendingFlowchartNodes` with try/catch. Coordinate-space: single `scale(zoom)`; `renderElement` and `frameClip` apply scroll internally. Ordering invariant: non-iframe elements first, then iframes/embeddables on top, then flowchart ghosts.
- `renderStaticSceneThrottled` (L482-L486) — `throttleRAF`-wrapped `_renderStaticScene` ("throttled to animation framerate" per L481).
- `renderStaticScene(renderConfig, throttle?)` (L491-L501) — public entry; throttled or synchronous depending on `throttle`.

Notable cross-file performance pattern: both the static and new-element scenes use `throttleRAF` so high-frequency state changes collapse to one paint per frame; the interactive scene is throttled by its caller (`Renderer`) instead and additionally returns animation state so the `AnimationController` can keep re-invoking it for the animated binding highlight.
