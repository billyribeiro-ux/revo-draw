## Cluster: excalidraw-app__(root)__1

This cluster covers three small, infrastructural files of the Excalidraw web application: Sentry error-reporting initialization, the app theme React hook, and the Vite environment type declarations. None of these touch canvas geometry directly; the theme hook is the only file with parity-relevant behavior for a reimplementation.

### excalidraw-app/sentry.ts

Purpose: Side-effecting module that initializes the Sentry browser error-reporting SDK, gating it by deploy environment and registering a feature-flag integration; importing the module performs the init.

This file declares no exported functions. It executes top-level statements and one inline callback. Significant constants and the inline function:

- `SentryEnvHostnameMap: { [key: string]: string }` (L5-L9) — Maps known hostnames to Sentry `environment` labels: `excalidraw.com` → `"production"`, `staging.excalidraw.com` → `"staging"`, `vercel.app` → `"staging"`. Used to both detect whether the app is running in an "online" deploy and to label the environment.
- `SENTRY_DISABLED = import.meta.env.VITE_APP_DISABLE_SENTRY === "true"` (L11) — Build-time/env kill switch read from Vite env; truthy when the env var string equals `"true"`.
- `onlineEnv` (L14-L18) — Computed string-or-undefined. Evaluates to `false`/`undefined` when Sentry is disabled, otherwise the first key in `SentryEnvHostnameMap` whose substring appears in `window.location.hostname` (via `hostname.indexOf(item) >= 0`). Side effect: reads `window.location.hostname` at module load — requires a browser/DOM. Invariant: substring match, so any subdomain containing `vercel.app` is treated as staging.
- `Sentry.init({...})` (L20-L83) — Initializes Sentry. `dsn` is the real DSN string only when `onlineEnv` is truthy, else `undefined` (so SDK installs but reports nothing locally/Docker, preserving privacy). `environment` is the mapped label or `undefined`. `release` comes from `import.meta.env.VITE_APP_GIT_SHA`. `ignoreErrors` (L26-L32) is a list of known-noise error strings/regexes: Safari `window.__pad.performLoop`, IndexedDB transaction-on-closing error, dynamic-import/fetch failures from stale service-worker assets, `QuotaExceededError` for localStorage, and IndexedDB backing-store open failures. `integrations` (L33-L38) enables `captureConsoleIntegration` for `error` level only plus `featureFlagsIntegration`.
  - `beforeSend(event)` (L39-L82) — Inline event-mutating hook. (1) Strips the URL fragment from `event.request.url` via `replace(/#.*$/, "")` (L40-L42) to avoid leaking hash-encoded scene data. (2) When the event has no `exception` (i.e. it came from a captured `console.error`, not a thrown error), synthesizes an exception payload (L44-L79): sets `type: "ConsoleError"`, `value` from `event.message` (fallback `"Unknown error"`), and builds a `stacktrace.frames` array by walking `callsites()` (from the `callsites` package), dropping the first frame (`.slice(1)`), filtering out frames with no filename and frames inside `@sentry_browser.js`, and mapping each remaining frame to `{ filename, function, in_app, lineno, colno }`. `in_app` is `true` unless the filename includes `node_modules`. A `mechanism` block marks it as an instrumented, handled `console.error` capture. Returns the (mutated) `event`. Non-obvious detail: this manual stack synthesis exists because console-captured errors lack a native stack; the `.slice(1)` drops the Sentry wrapper frame.
- Feature-flag registration (L85-L94) — After init, fetches the `FeatureFlags` integration via `Sentry.getClient()?.getIntegrationByName<Sentry.FeatureFlagsIntegration>("FeatureFlags")`; if present, calls `addFeatureFlag("COMPLEX_BINDINGS", getFeatureFlag("COMPLEX_BINDINGS"))` so the flag value is attached to subsequent Sentry events. Side effect / dependency: `getFeatureFlag` from `@excalidraw/common`.

### excalidraw-app/useHandleAppTheme.ts

Purpose: Defines the `useHandleAppTheme` React hook that manages the app's theme tri-state (`light`/`dark`/`system`), resolving it to a concrete editor theme, persisting the choice to localStorage, reacting to OS color-scheme changes, and toggling via a keyboard shortcut.

- `getDarkThemeMediaQuery(): MediaQueryList | undefined` (L9-L10) — Returns `window.matchMedia?.("(prefers-color-scheme: dark)")`, or `undefined` when `matchMedia` is unavailable (SSR/old browsers). Pure aside from reading `window`.
- `useHandleAppTheme = () => { editorTheme, appTheme, setAppTheme }` (L12-L70) — Exported React hook. State and behavior:
  - State `appTheme` via `useState<Theme | "system">` (L13-L20) — Lazily initialized from `localStorage.getItem(STORAGE_KEYS.LOCAL_STORAGE_THEME)` cast to `Theme | "system" | null`, falling back to `THEME.LIGHT`. This is the user's persisted preference (may be `"system"`).
  - State `editorTheme` via `useState<Theme>(THEME.LIGHT)` (L21) — The resolved concrete theme (`light` or `dark`) actually handed to the editor; never `"system"`.
  - `useEffect` (L23-L55), deps `[appTheme, editorTheme, setAppTheme]` — Sets up two listeners. (a) `handleChange(e: MediaQueryListEvent)` (L26-L28) sets `editorTheme` to dark/light based on `e.matches`; this listener is attached to the dark-scheme media query **only when `appTheme === "system"`** (L30-L32). (b) `handleKeydown(event: KeyboardEvent)` (L34-L45) listens (capture phase, on `document`) for the shortcut **Alt+Shift+D without Ctrl/Cmd** (`!event[KEYS.CTRL_OR_CMD] && event.altKey && event.shiftKey && event.code === CODES.D`); on match it `preventDefault()` + `stopImmediatePropagation()` and toggles `appTheme` to the opposite of the **current resolved `editorTheme`** (dark→light else dark). Cleanup (L49-L54) removes both listeners. Notable: the toggle reads `editorTheme` (resolved), not `appTheme`, so toggling while in `"system"` mode flips to the explicit opposite of what the OS currently shows.
  - `useLayoutEffect` (L57-L67), deps `[appTheme]` — Persists `appTheme` to `localStorage` under `STORAGE_KEYS.LOCAL_STORAGE_THEME`, then resolves `editorTheme`: if `appTheme === "system"` it reads `getDarkThemeMediaQuery()?.matches` to pick dark/light, otherwise it sets `editorTheme = appTheme`. Runs synchronously before paint to avoid a theme flash.
  - Returns `{ editorTheme, appTheme, setAppTheme }` (L69) — Consumers render with `editorTheme`, display/select with `appTheme`, and change preference via `setAppTheme`.
  - Invariants: `editorTheme` is always a concrete `Theme`; `appTheme` is the only value persisted; the media-query change listener is active only in `"system"` mode. Side effects: localStorage write, two global event listeners (one conditional). Dependencies: `THEME` from `@excalidraw/excalidraw`; `EVENT`, `CODES`, `KEYS` from `@excalidraw/common`; `STORAGE_KEYS` from `./app_constants`; `Theme` type from `@excalidraw/element/types`.

### excalidraw-app/vite-env.d.ts

Purpose: TypeScript ambient declaration file (types only) that adds Vite PWA/SVGR triple-slash references and types the application's `import.meta.env` environment variables.

This file contains no executable code or functions. It consists of:

- Three triple-slash reference directives (L1-L3): `vite-plugin-pwa/vanillajs`, `vite-plugin-pwa/info`, `vite-plugin-svgr/client` — pull in ambient types for the PWA and SVGR Vite plugins.

(Note: line 11's comment for `VITE_APP_WS_SERVER_URL` has an unterminated `(https:` — a stray comment artifact, harmless.)
- `interface ImportMeta` (L47-L49) — Augments `import.meta` with `readonly env: ImportMetaEnv`.
