## Cluster: excalidraw__subset__1

This cluster contains the WOFF2 font compression/decompression WASM integration layer: a small lazy-loading wrapper module and the large generated, base64-embedded WASM binary. These are part of a separately lazy-loaded chunk (the loader module is explicitly written to avoid importing anything from the main chunk to prevent the whole main chunk being pulled into the lazy chunk). WOFF2 (de)compression here is used in font subsetting ‚Äî see the sibling `subset/` files (e.g. `subset-main.ts`, `woff2-bindings.ts`) that import this layer.

### packages/excalidraw/subset/woff2/woff2-loader.ts

Lazily instantiates the WOFF2 WASM module (passing the embedded binary), and exposes thin `compress`/`decompress` functions that convert the emscripten-bound C++ `std::vector` results into `Uint8Array`, memoizing the initialization so the WASM is loaded at most once.

- `convertFromVecToUint8Array(vector: Vector): Uint8Array<ArrayBuffer>` ‚Äî L21-L28. Internal helper. Iterates an emscripten-bound C++ vector via its `vector.size()` and `vector.get(i)` accessors, pushing each byte into a plain JS array, then constructs a `Uint8Array` from it. Input: an opaque emscripten `Vector` (typed as `any` via the local `type Vector = any` alias on L16). Output: a byte array copy. Side effects: none. Notable: it materializes the entire vector into a JS array first (an O(n) extra allocation) rather than reading the WASM heap directly ‚Äî a potential perf/copy cost for large font buffers, relevant if a reimplementation wants to avoid the double copy.
- `load(): Promise<{ compress: (buffer: ArrayBuffer) => Uint8Array<ArrayBuffer>; decompress: (buffer: ArrayBuffer) => Uint8Array<ArrayBuffer>; }>` ‚Äî L31-L67 (const arrow). Internal. Wraps emscripten module initialization in a manually-constructed `Promise`. Calls the imported `bindings({ wasmBinary: binary })` factory (from `./woff2-bindings`, passing the embedded `binary` from `./woff2-wasm`) so the runtime uses the inlined WASM rather than fetching a `.wasm` URL. On resolution it receives a `module` exposing `woff2Enc(buffer, byteLength)` and `woff2Dec(buffer, byteLength)`, and builds a small `woff2` object whose `compress`/`decompress` call those C++ entry points (passing `buffer` and `buffer.byteLength`) and pipe the returned vector through `convertFromVecToUint8Array`. Notable invariant / footgun documented in-code (L37-L38, L44-L45): emscripten's `bindings(...).then` is **not** Promise/A+ compliant, so an explicit nested `try/catch` is used inside the `.then` callback (L44-L60) in addition to the outer `try/catch` (L36-L65); both reject the outer promise on error. Re-exporting only `compress`/`decompress` is also deliberate to "avoid an infinite loop inside emscripten bindings" (L45). Side effects: instantiates the WASM module.
- `default export: (): ReturnType<typeof load>` ‚Äî L70-L76. The public lazy entry point. Memoizes via the module-level `loadedWasm` variable (L18, typed `ReturnType<typeof load> | null`): on first call it invokes `load()` and caches the resulting promise; subsequent calls return the same cached promise. Invariant: WASM is initialized at most once per module lifetime; concurrent callers share one in-flight promise. Side effect: sets `loadedWasm`.

Notable module-level details: the file header comment (L1-L8) is a hard architectural constraint ‚Äî this module must not import from the main chunk (even tree-shakeable imports leak the whole chunk in dev), which is why it duplicates trivial logic instead of reusing shared utils. There is a `TODO` (L30) noting future support for fetching the WASM from a URL/CDN/data-URL instead of inlining. The `Vector` type is a local `any` alias (L16). No `convert-to-vector` direction exists ‚Äî input `ArrayBuffer`s are passed straight to the C++ functions; only outputs are converted.

### packages/excalidraw/subset/woff2/woff2-wasm.ts

Generated, do-not-edit module that ships the WOFF2 codec WASM binary inlined as a base64 string and decodes it to a `Uint8Array` at import time via a fast inline base64 decoder; it is the `binary` consumed by `woff2-loader.ts`.

- File banner / provenance ‚Äî L1-L3 mark it `GENERATED CODE -- DO NOT EDIT!`, `eslint-disable`, and `@ts-nocheck`. L5-L34 is a doc comment recording that the WASM is produced by `scripts/buildWasm.js`, that the source is the `fonteditor-core` npm package (authors kekee000 / junmer, MIT, v2.4.0), and embeds the full MIT license text.
- `__toBinary: (base64: string) => Uint8Array` ‚Äî L37-L53. An IIFE-created, `/* @__PURE__ */`-annotated faster `atob` alternative (the comment L37 cites the esbuild issue it was taken from). Initialization (L38-L41): builds a 128-entry `Uint8Array` lookup `table` mapping base64 character codes to their 6-bit values using the branchless index expression `table[i < 26 ? i + 65 : i < 52 ? i + 71 : i < 62 ? i - 4 : i * 4 - 205] = i` for `i` in `[0,64)` (covers `A‚ÄìZ`, `a‚Äìz`, `0‚Äì9`, and the `+`/`/` chars). Decode (L42-L52): computes output length as `(n - (base64[n-1]=="=") - (base64[n-2]=="=")) * 3 / 4 | 0` to account for `=` padding, then walks the input four base64 chars at a time, looking up `c0,c1,c2,c3` and emitting three bytes via the standard bit recombination `c0<<2|c1>>4`, `c1<<4|c2>>2`, `c2<<6|c3`. Returns the decoded `Uint8Array`. Notable: no validation of input characters (assumes well-formed generated base64); performance-oriented bit-twiddling rather than `atob`+`charCodeAt`. Side effects: none (pure).
- `default export` ‚Äî L55. `export default __toBinary(\`AGFzbQEAAAAB‚Ä¶\`);` ‚Äî a single ~969,618-character base64 template literal (the WASM module bytes) decoded eagerly at import to a `Uint8Array`. This is the `binary` imported by `woff2-loader.ts` (L11) and handed to emscripten as `wasmBinary`. The leading `AGFzbQEAAAAB` decodes to the WASM magic `\0asm` + version 1, confirming it is a raw WebAssembly binary. Side effect: a ~700KB+ `Uint8Array` allocation at module-load time; this is why the file lives in its own lazy chunk.

Performance / parity note for a reimplementation: the WOFF2 codec is a WASM blob (~700KB decoded) loaded lazily and memoized. A Svelte/Canvas parity port would either (a) load WOFF2 (de)compression via a separately code-split chunk and a one-time-init promise like this loader, or (b) fetch the `.wasm` from a URL (the `TODO` path) to avoid inlining ~950KB of base64 into a JS bundle. The output-conversion path copies vector‚ÜíJS array‚Üí`Uint8Array` (two allocations); reading the WASM heap directly would be faster.
